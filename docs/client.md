# Client architecture (React + Vite)

Read before frontend work under `client/`.

## API communication
- `getApiHost()` (`client/src/utils/api/get-api-host.util.ts`) — returns `VITE_API_URL` in dev, or the hardcoded Render URL in production
- Vite proxies `/api` → `http://localhost:3000` in dev (configured in `vite.config.ts`)
- `useAuthenticatedAxios()` — hook that returns an Axios caller pre-configured with `Authorization: Bearer <token>`
- All data-fetching hooks live under `client/src/hooks/api/` and use TanStack Query

## Hooks pattern
Each domain under `client/src/hooks/api/` has separate `.query.ts` and `.mutation.ts` files. Queries use `useSuspenseQuery` or `useQuery`; mutations use `useMutation` with `onSuccess` invalidating related queries. Query keys follow a hierarchical array pattern: `["resource", ...filters]` (e.g., `["moodle-links", authUserId]`). Errors are typed as `AxiosError<ApiErrorDto>`.

## Auth flow
`AuthProvider` (`client/src/providers/auth/auth.provider.tsx`) persists `{ token, user }` to `localStorage` under the key `"userInfo"`. On mount it reads it back via `readUserInfo()`. `useAuthInfo()` exposes the context; `useRole()` extracts the role. Routes check role to show/hide menu items. All protected routes are guarded on the server side by the global `AuthGuard`. Use `<AuthzHide roles={[Role.ADMIN]}>` (`client/src/components/permissions/authz-hide.tsx`) to conditionally render UI elements based on role.

## Routing
`client/src/router.tsx` defines all routes and the top-level layout. The sidebar is role-aware: Reports is visible to `admin`, `manager`, and `viewer`; the Administración sub-menu is only visible to `admin`. Administración has Organization plus four category entries that each open a card page (no nested sub-menus): **Importaciones** (`/tools/importaciones`), **Gestión y acceso** (`/tools/gestion-acceso`), **Correo** (`/tools/correo`) and **Herramientas** (`/tools/herramientas`). Each page renders `components/tools/ToolList.tsx` with a `categoryKey` prop, filtering `toolCategories` to that group's tool cards; `/tools` with no key still shows every category. Tool definitions and their grouping live in `toolCategories` (Importaciones: Moodle, data cross-reference, SAGE, INAEM · Gestión y acceso: user management, audit log · Correo: SMTP settings `/organization/smtp`, email log · Herramientas: forum duplicator).

**Responsive sidebar:** breakpoint is `md` (768 px). On `md+` a persistent `<Sider>` is rendered. Below `md` the Sider is replaced by a `<Header>` with a hamburger button that opens a `<Drawer>`. Use `screens.md === false` (not `!screens.md`) to detect mobile — avoids a flash on first render when `useBreakpoint` hasn't resolved yet. Drawer closes when any leaf `<Link>` is clicked; the Administración sub-menu can expand/collapse without closing the drawer because `onClick={onClose}` is placed on the `<Link>` nodes, not on the `<Menu>`.

## Responsive design conventions
**Form layouts:** all entity forms (create + detail for users, centers, companies, courses, groups) use `<Row gutter={[16, 0]}>` + `<Col xs={24} sm={X} md={Y}>` — never `<div style={{ display: 'flex', gap: 16 }}>` and never hardcoded pixel/`ch` widths on `<Form.Item>` (let the `<Col>` control the width). `<DatePicker>` and `<Select>` inside a Col need `style={{ width: '100%' }}` to fill their column. Button bars at the end of a form use the shared `.form-actions` class (`index.css`) instead of an inline flex `<div>` — it wraps to multiple lines on mobile. Forms are React Hook Form + Zod (`Controller` wrapping the Ant input) — keep that pattern.

**Wide tables:** add `scroll={{ x: 'max-content', y: N }}` and pin key columns with `fixed: 'left'`. Fixed columns must be contiguous from the left edge.

**Table navigation:** every navigation table — main lists (`users`, `groups`, `courses`, `companies`, `centers`) **and** navigation-only sub-tables inside detail pages (center-detail Usuarios, `user-courses-section`, `user-preinscriptions-section`) — uses the shared **`<DataTable>`** with a `getRowUrl(record)` prop: single click opens that URL in a **new tab** (the app-wide navigation gesture). The **only** exception is tables with **row selection** (checkboxes: `GroupUsersManager`, `CreateUserGroupModal`), which keep `onDoubleClick` to open because single click already toggles selection. Don't reintroduce `navigate()` on rows or scatter `window.open` — go through `getRowUrl` (or `openDetail` for non-table links). `getRowUrl` may return a path with query params (e.g. `user-courses-section` builds `/courses/:id?groupId=&userId=`).

## Design system & shared list primitives
Single source of truth, introduced to unify the listings (read before adding a new list or status tag):
- **Theme:** `theme/tokens.ts` holds the global `ConfigProvider` config (brand color, table density, `MOBILE_BREAKPOINT`); wired in `main.tsx`. `hooks/use-is-mobile.ts` reads it (`screens.md === false`).
- **Semantic colors:** `theme/semantic-colors.ts` is the **only** place tag colors live. `STATUS_COLORS` holds the "semáforo" **primitives** (`active`/`inactive`/`warning`/`processing`/`neutral`) — every recurring state colour (tags **and** the admin-tool status maps like `getStatusColor`/`getConfidenceColor`/`getActionColor`) reads from it, so changing a primitive propagates everywhere. `CLIENT_COLORS`/`FUNDING_COLORS`/`MODALITY_COLORS` cover the categorical axes. Never pass a literal `color="green"`/`"success"` for a *status* — use `STATUS_COLORS` or a tag component. Purely categorical/decorative colours (a blue chip for "CSV source", HTTP-method colours in the audit log) may stay as literals. `modalityColor` is case-insensitive (`PRESENCIAL` and `Presencial` both resolve).
- **Tags:** `components/common/tags/` exports `<ClientTag>`, `<FundingTag>`, `<ModalityTag>`, `<ActiveTag>` (optional `title` tooltip), `<ProvisionalTag>`, `<FinalizedTag>` (green "Finalizado" / red "No finalizado"), `<BajaTag>`. Use these for the recurring states instead of re-implementing `<Tag color=…>`.
- **List building blocks:** `components/common/DataTable.tsx` (Ant Table wrapper: horizontal scroll for mobile, density, `getRowUrl` navigation, optional `scrollY`), `components/common/ListPageLayout.tsx` (responsive controls bar via the `.list-controls` CSS class + `useTableScroll` wiring), `hooks/use-list-pagination.ts` (server-side pagination config) and `hooks/use-list-search.ts` (search + debounce + `normalizeSearch`).
- **Feedback (message / modal / notification):** **always** `const { message, modal, notification } = App.useApp();` — the single standard across the app. The global `<App>` provider is mounted in `main.tsx`, so there is **no** manual `contextHolder` to render. Do **not** use the static `message.*` / `Modal.confirm` (they ignore the ConfigProvider theme), nor the local-holder hooks `message.useMessage()` / `Modal.useModal()` / `notification.useNotification()`. Delete confirmations go through `modal.confirm({ okText, okType: "danger", cancelText, onOk })`. In **tests**, components that call `App.useApp()` must render inside antd `<App>` (or mock `App.useApp` to return `{ message, modal }`), otherwise the returned api is undefined.
- **Page tabs:** `components/common/RouteTabs.tsx` is the single tab pattern for detail pages. Semantic string keys (`datos`, `usuarios`, `centros`, `moodle`, `cursos`, `preinscripciones`, `ficha`, `contenidos`) synced to `?tab=` via `useSearchParams({ replace })`; the default tab is rendered **without** the param (clean URLs). Conditional tabs work (valid keys are derived from `items` each render). Don't use numeric `"1"/"2"` keys or hand-rolled `navigate()` URL sync. When a tabbed component is **embedded** inside another tabbed page (e.g. `UserDetail` in the course-detail lookup modal), pass `syncUrl={false}` (UserDetail exposes `syncTabsToUrl`) so the two don't fight over `?tab=`. **Deep links:** the company-detail Centros tab is `?tab=centros` (renamed from the old `?tab=centers`).
- **Formatting & navigation:** `utils/format.ts` (`formatDate`/`formatDateTime`/`formatNumber`, es-ES) and `utils/open-detail.ts` (`openDetail(path)` → new tab). Client-side filtering uses `normalizeLoose` + `matchesLoose` from `utils/normalize-search.ts` (the per-route inline `normalize` helpers were removed).

**Baja flag (`B`):** `components/common/tags/baja-tag.tsx` exports `<BajaTag user={user} />` (the old `components/users/baja-tag.tsx` path is kept as a re-export for compatibility). Renders a small red `B` tag next to the user's name when the user's **centro principal** (`is_main_center`) has an `end_date` (fecha de baja); renders nothing otherwise. The helper reads `user.centers[]` (the main list, center-detail Usuarios tab, and group listings all return this enriched array), with a `user.main_center_end_date` fallback for any payload that only carries the main center's baja date. Wired into the Nombre column of `users.route.tsx`, `centers/center-detail.route.tsx`, and the shared `USERS_TABLE_COLUMNS` (group members + bonification modal).

A **"Mostrar dados de baja"** checkbox (default checked) in the main users list and the center-detail Usuarios tab toggles `useUsersQuery({ includeInactive })`. Filtering is **server-side** (lists are paginated): `includeInactive: false` sends `include_inactive=false`, which adds a `NOT EXISTS` over the user's main center with `end_date` in `UserRepository.findAllPaginated` — i.e. it hides exactly the users that carry the `B` flag.

The center-detail Usuarios tab also has a **"Solo centro principal"** checkbox (default checked) → `useUsersQuery({ mainCenterOnly })`. When set it sends `main_center_only=true`, which (combined with `id_center`) tightens the center `EXISTS` with `AND uc.is_main_center = true`, so only users whose **main** center is this one are listed (vs. any membership). No-op without `id_center`.

**Cross-page preselection via URL params:** single-clicking a course row in the user detail "Cursos" tab (`user-courses-section.tsx`, via `DataTable` `getRowUrl`) opens `/courses/:id?groupId=&userId=` in a new tab (no shared React Query cache between tabs). `course-detail.route.tsx` reads `groupId`/`userId` via `useSearchParams` once on mount to preselect the group (falls back to the default first group if not found) and highlight the user (`GroupUsersManager`'s `highlightUserId` prop, applied as a selected checkbox + scroll-into-view, not a CSS highlight).

**Gotcha — refs inside `setState` updaters:** never mutate a `ref` inside a functional `setState` updater (`setX((prev) => { ref.current = ...; return ...; })`). React 18 `<StrictMode>` invokes updater functions twice to detect impurities; the second call sees the ref already mutated and silently produces the wrong result. Consume/mutate the ref in the effect body instead, guarded by a dedicated "applied" boolean ref (refs are safe across StrictMode's double-invocation since they persist between calls).

## Course typology (UI)
Data model / enums → `docs/architecture.md`. UI consumption of the axes (modality / `client` / `funding`; ámbito público-privado is derived from `funding`):

`utils/course-profile.ts` (`getCourseProfile({ modality, client, funding })`) is the **single source of truth** for type-driven UI — returns axis flags (`isInaem` from `client`; `isPublic`/`isPrivate` derived from `funding`) and capability flags (`showMoodleSync`, `showProgressColumn`, `showFinalizedColumn`, `showBonificationButton`, `showExpediente`, `showPreinscripciones`). Use it instead of scattering `modality === 'presencial'` checks. `showBonificationButton` is permissive (hidden only for explicit non-FUNDAE funding, matching the server guard).

- **Group members table** (`GroupUsersManager`, the main `getCourseProfile` consumer): columns adapt to the course type.
  - `showFinalizedColumn` = presencial **or** INAEM (any modality). Presencial: the **Finalizado** column (tag green/red from `user_group.finalized`) *replaces* **Porcentaje** and drops Moodle/Tiempo (no Moodle progress). INAEM online/mixta: **Finalizado** is *appended* while keeping the online columns (Moodle sync + progreso).
  - **Porcentaje, Tiempo usado y Finalizado** only render a value for students (`isStudentUser`); other roles (tutores, etc.) show an empty cell. Applies to every course type.

- **Courses list** (`courses.route.tsx`): `Segmented` tabs by **funding** — **Todos / Pública / FUNDAE / Privada / Sin clasificar** (client-side predicates over the loaded list; server-side `FilterCourseDTO` exists for API use). The **client** (INAEM/VITALIA/OTRO) is a **Cliente column with an Ant table filter**, not a tab — so the 4 buckets are reachable by tab×filter (e.g. *Pública*+INAEM, *FUNDAE*+VITALIA). Active tab in URL (`?tab=`); columns adapt per tab.
- **Course form** (create + detail): **Cliente** select (INAEM/VITALIA/OTRO) + **Financiación** select; **Nº Expediente** only for `client=INAEM`, **FUNDAE ID** only for FUNDAE funding (detail keeps them visible if a value already exists).
- **Users list** (`users.route.tsx`): **Tipo de formación** select (`fundae`/`publica`/`inaem`/`privada`) — *derived* filter (no type column on `user`); server-side `FilterUserDTO.formation_type` via `EXISTS` over `user_group → groups → courses` (`formation-type.util.ts` maps each value to a `funding`/`client` predicate). `inaem` filters by `client`; the rest by `funding`.

## Type sharing
Shared types between components and hooks live in `client/src/shared/types/`. Component-level types stay in `client/src/types/`. Zod schemas for form validation are in `client/src/schemas/` (currently only Spanish DNI and CIF validators; form schemas are co-located with their route components).

## Tests
Vitest with `happy-dom` environment. Test files are co-located with routes/components as `*.spec.tsx`. Type-check the whole client with `npx tsc -b --noEmit`.
